# Plan: Switch `builder.sh` to a single `native-multi` buildx builder

## Goal
Replace the current two-builder, two-build, two-tag + manifest-merge flow in `builder.sh` with a single multi-arch build that uses the existing `docker buildx` context/endpoint named `native-multi`. The result is one push that produces a single multi-arch image tag covering both `linux/amd64` and `linux/arm64`.

## Context (current state)

`builder.sh` (62 lines) currently does:

1. Defines two builder env vars:
   - `AMD64_BUILDER` (default `amd64`)
   - `ARM64_BUILDER` (default `arm64-host`)
2. Inspects both builders.
3. Runs `docker buildx build --builder "$AMD64_BUILDER" --platform linux/amd64 ... --push` with two amd64-suffixed tags.
4. Runs `docker buildx build --builder "$ARM64_BUILDER" --platform linux/arm64 ... --push` with two arm64-suffixed tags.
5. Calls `docker buildx imagetools create` to combine the two per-arch tags into a multi-arch manifest at `${IMAGE_BASE}:${IMAGE_TAG}` and `${IMAGE_BASE}:${VERSION_TAG}`.
6. Inspects and prints the manifest.

User requirement: Use a single buildx context/endpoint named `native-multi` (already configured on this host with both arch nodes attached) so a single `docker buildx build --platform linux/amd64,linux/arm64 --push` produces one multi-arch tag in one shot.

User decisions (already confirmed):
- **Single multi-arch tag only** — do not keep `-amd64` / `-arm64` suffixed tags.
- **Minimal change** — keep all existing env-var overrides, just collapse the builder vars into one `BUILDER=native-multi` and drop the manifest-merge step.

Other places that reference arch/buildx (no changes required, just noted):
- `.github/workflows/build-deploy-container.yml` — uses GitHub Actions `setup-buildx-action`, unrelated to local script.
- `.github/workflows/build-deploy-develop-container.yaml` — same.
- `docker-compose.prod.yml` — references `bigcapitalhq/webapp:latest` etc., not affected.

## Changes

### Edit `builder.sh`

Single file edit. Replace the two-builder/two-build/manifest flow with a single multi-arch build.

**Config section** — collapse builder vars:

```bash
# ---- config ----
# Single buildx context/endpoint that has both amd64 and arm64 nodes attached
BUILDER="${BUILDER:-native-multi}"

IMAGE_BASE="${IMAGE_BASE:-ghcr.io/mestre8d/bigcapital-docker}"
IMAGE_TAG="${IMAGE_TAG:-webapp}"
VERSION_TAG="${VERSION_TAG:-latest}"
DOCKERFILE="${DOCKERFILE:-./packages/webapp/Dockerfile}"
CONTEXT_DIR="${CONTEXT_DIR:-.}"
PLATFORMS="${PLATFORMS:-linux/amd64,linux/arm64}"

# Final multi-arch image references
IMAGE_REF="${IMAGE_BASE}:${IMAGE_TAG}"
VERSION_REF="${IMAGE_BASE}:${VERSION_TAG}"
```

**Sanity check** — only inspect the single builder:

```bash
docker buildx inspect "${BUILDER}" >/dev/null
```

**Build & push** — single `buildx build` with both platforms and both tags:

```bash
echo "==> Building ${PLATFORMS} via builder '${BUILDER}' and pushing ${IMAGE_REF} / ${VERSION_REF}"
docker buildx build \
  --builder "${BUILDER}" \
  --platform "${PLATFORMS}" \
  --file "${DOCKERFILE}" \
  --tag "${IMAGE_REF}" \
  --tag "${VERSION_REF}" \
  --push \
  "${CONTEXT_DIR}"
```

**Inspect** — verify the resulting manifest is multi-arch:

```bash
echo "==> Inspecting manifest"
docker buildx imagetools inspect "${VERSION_REF}"
```

**Final summary** — only the two pushed tags:

```bash
echo "Done."
echo "Pushed:"
echo "  ${IMAGE_REF}"
echo "  ${VERSION_REF}"
```

### Removals
- `AMD64_BUILDER`, `ARM64_BUILDER` env vars.
- `AMD64_IMAGE`, `ARM64_IMAGE`, `MANIFEST_IMAGE` derived names (no longer needed).
- The two separate `docker buildx build` invocations.
- The `docker buildx imagetools create` step (single build with two tags already produces both pointer tags into the multi-arch manifest).
- The four `-amd64` / `-arm64` suffixed tags.

### Notes / preserved behavior
- `IMAGE_TAG` and `VERSION_TAG` are still both pushed (current script also pushes both). With a single `buildx build --tag X --tag Y --push`, both tags point at the same multi-arch index, so behavior matches the old manifest output.
- Default `BUILDER=native-multi` matches the user's described context. Override by exporting `BUILDER=...` before invoking.
- Default `PLATFORMS=linux/amd64,linux/arm64` is overridable in case the user later wants a single arch.
- `set -euo pipefail` and the `#!/usr/bin/env bash` shebang are kept.
- File mode (executable) is unchanged — `builder.sh` is currently `-rw-r--r--` in the repo but that's how the user already invokes it (likely via `bash builder.sh`); no chmod change planned.
